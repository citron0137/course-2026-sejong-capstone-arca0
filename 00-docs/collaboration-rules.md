## 1. GitHub 사용 규칙

### 1-1. 깃 브랜치 전략

- **작업 단위 브랜치 생성**  
  - 맡은 작업은 **작업 단위로 별도 브랜치**를 만들어 진행한다.
  - 작업 단위는 가능한 **작고 명확하게** 나눈다.  
    - 예시: `feature/login-ui`, `fix/player-move-bug`, `chore/update-readme`
- **브랜치 이름 규칙(권장)**  
  - `타입/간단-설명` 형식 사용  
    - 타입 예시: `feature`, `fix`, `chore`, `refactor`, `docs` 등

### 1-2. 작업 완료 후 반영 프로세스
- **1) main 기준으로 리베이스(rebase)**  
  - PR 생성 전, 작업 브랜치에서 최신 `main`을 기준으로 리베이스 한다.
  - 충돌이 발생하면 **브랜치에서 직접 해결**하고, 로컬에서 빌드/테스트까지 확인한다.

- **2) PR(Pull Request) 생성**  
  - PR 제목은 **무엇을 하는 PR인지 한 줄 요약**으로 작성한다.  
    - 예시: `feat: 로그인 UI 구현`, `fix: 캐릭터 이동 속도 버그 수정`
  - PR 본문에는 다음 내용을 포함하는 것을 권장한다.
    - 주요 변경 사항 요약
    - 테스트 여부 및 방법
    - 리뷰어가 특히 봐줬으면 하는 부분

- **3) 리뷰 & APPROVE**  
  - 최소 1명 이상의 팀원이 코드를 리뷰하고 **APPROVE**를 남긴다.
  - 리뷰어가 수정 요청을 하면, 작업 브랜치에서 수정 후 해당 PR에 계속 푸시한다.

- **4) squash merge 적용**  
  - PR이 승인되면 **squash merge**로 `main`에 반영한다.
  - squash 시 최종 커밋 메시지는 PR의 목적이 잘 드러나도록 정리한다.

### 1-3. 기타 권장 사항
- **커밋 단위는 작게, 메시지는 명확하게**  
  - 하나의 커밋에는 하나의 의미 있는 변경만 포함하도록 노력한다.
- **main 브랜치 직접 푸시 금지(원칙)**  
  - 모든 변경은 PR 과정을 통해 `main`에 반영하는 것을 원칙으로 한다.

## 2. 작업 나누기 규칙

- **레이어별 작업 분리**
  - 작업은 다음 다섯 가지 영역/레이어로 **명확히 분리**해서 진행한다.
    - **도메인 개발**: 핵심 비즈니스 로직, 엔티티, 도메인 서비스
    - **애플리케이션 레이어 개발**: 유즈케이스, 서비스 조합, 워크플로 제어
    - **UI 레이어 개발**: 화면, 입력/출력, 사용자 인터랙션
    - **인프라 개발**: DB, 네트워크, 파일 시스템, 외부 API 연동 등
    - **에셋 작업**: 아트, 사운드, 애니메이션, 이펙트 등 리소스 제작 및 세팅
  - 하나의 작업/브랜치는 가능하면 **하나의 레이어(또는 에셋 영역)에만 집중**하도록 나눈다.

- **인터페이스 선(先) 정의 원칙**
  - 새로운 기능을 개발하기 전에, **레이어 간에 필요한 인터페이스를 먼저 정의**한다.
    - 예시: 도메인에서 사용할 `Repository` 인터페이스, 애플리케이션에서 사용할 `Service` 인터페이스, UI에서 호출할 `UseCase` 인터페이스 등
  - 인터페이스가 합의된 이후에, 각 레이어 담당자가 **병렬로 구현을 진행**한다.
  - 인터페이스 변경이 필요할 경우, 관련된 사람들과 **사전 합의 후에 수정**한다.
